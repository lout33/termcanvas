# Canvas File Preview Design

## Goal

Add a simple file-observer workflow around the existing terminal canvas so users can inspect what agents and shells are creating without turning the app into an editor or IDE.

## Product Intent

This app remains canvas-first.

Terminal nodes are still the primary way work happens. The new folder and file affordances exist only to help the user inspect outputs created by those terminals.

That means the first version should stay read-only, lightweight, and visually secondary to the canvas itself.

## User Story

As a user, I want to browse folders and preview markdown, JSON, and text/code files generated by terminal nodes so I can quickly inspect what agents are doing without leaving the app.

## Scope

In scope:

- show a collapsible folder tree in the left `Folders` section
- show a read-only file preview in a right-side inspector
- support markdown, JSON, and text/code preview
- support manual refresh of the selected file preview
- keep existing workspace folder selection behavior
- keep existing canvas import/export and terminal behavior working

Out of scope:

- file editing
- binary/media preview beyond a fallback message
- preview nodes on the canvas
- syntax highlighting beyond basic readable text presentation
- live auto-refresh watchers for the selected preview file
- multiple simultaneous file previews

## UX Design

### Left Sidebar

The left panel keeps two sections:

- `Canvases`
- `Folders`

The `Folders` section changes from a flat recursive list into a tree.

Behavior:

- clicking a directory row toggles expand/collapse
- expanded state is held in renderer state only
- clicking a supported file selects it and opens its preview in the right inspector
- unsupported files can still be selected, but the inspector shows a fallback message instead of content

The folder tree remains subordinate to the canvas. It should not visually dominate the screen or introduce editor-like chrome.

### Right Inspector

A new right-side inspector is added to the app shell.

It contains:

- file name
- relative path
- file type label
- `Refresh` button
- read-only preview body

Default states:

- no file selected: friendly empty state explaining that clicking a file previews it here
- loading: lightweight loading message
- error: clear read/read-path error message
- unsupported file: short fallback message
- too-large file: short fallback message

The right inspector is secondary UI. It should be readable but not visually heavier than the canvas.

## Supported Preview Rules

### Markdown

Markdown files are shown as raw source in a readable text view for the first version.

Reasoning:

- simplest implementation
- avoids introducing markdown rendering complexity and security concerns immediately
- still solves the core need of checking generated outputs

### JSON

JSON files are parsed when possible and then pretty-printed.

If parsing fails, the file is shown as raw text so malformed generated JSON is still inspectable.

### Text / Code

Text-like files are shown in a monospace read-only view.

The supported set should be extension-based and conservative. It should cover common generated outputs such as:

- `.txt`
- `.md`
- `.json`
- `.js`, `.ts`, `.tsx`, `.jsx`
- `.html`, `.css`
- `.py`, `.sh`
- `.yml`, `.yaml`

### Unsupported / Binary

If a file is not in the supported text-like set, the inspector should show a fallback message such as `Preview not available for this file type.`

### File Size Limit

Large files should not be fully loaded into the renderer preview.

The preview API should enforce a conservative byte cap and return a `tooLarge` result when exceeded. The UI should then show a message instead of partial unpredictable rendering.

## Architecture

### Main Process — `main.js`

Responsibilities:

- extend workspace tree generation to preserve parent/child structure suitable for folder expansion
- expose a new narrow IPC for reading a selected workspace file
- validate that the requested preview path stays inside the selected workspace root
- detect supported preview types and enforce a preview size limit
- return plain serializable preview payloads only

The main process remains the filesystem owner.

Proposed preview payload shape:

```js
{
  rootPath,
  relativePath,
  fileName,
  kind: "text" | "json" | "unsupported" | "too-large",
  language: "markdown" | "json" | "text" | "javascript" | ...,
  contents,
  lastModifiedMs
}
```

If the file cannot be read, the handler throws a clear error that the renderer turns into an error state.

### Preload — `preload.js`

Expose one new narrow method:

- `readWorkspaceFile(relativePath)`

No raw filesystem or arbitrary path access is exposed to the renderer.

### Renderer — `renderer.js`

Responsibilities:

- hold expanded-folder state
- hold selected file state
- hold preview loading, success, and error state
- render the collapsible folder tree
- request preview data from preload
- render the right inspector
- refresh the selected preview on demand
- clear invalid selection when workspace root changes or selected file disappears

The renderer should treat the tree and inspector as view state only. No filesystem trust decisions belong here.

### Markup / Styling — `index.html`, `styles.css`

The app shell gains a right inspector column.

High-level shell becomes:

- left sidebar: canvases + folders
- center workspace shell: board/canvas
- right inspector: selected file preview

The inspector should collapse gracefully when no workspace is open or no file is selected. It should not squeeze the canvas below usable size on ordinary laptop widths.

## Data Model Adjustments

The current workspace snapshot returns a flat list. For tree rendering, the renderer needs directory relationships.

Two acceptable options exist:

1. keep a flat list with relative paths and derive tree structure in the renderer
2. return a nested tree payload from main

Recommended choice: keep the transport flat and derive tree state in the renderer.

Reasoning:

- minimal change to the existing snapshot flow
- easier incremental migration from the current flat list UI
- preserves simple serializable payloads

Each entry should continue to include:

- `name`
- `relativePath`
- `kind`
- `depth`

The renderer builds parent-child display relationships from path segments and expanded state.

## Security Constraints

The file preview API must reject any preview request outside the selected workspace root.

Validation requirements:

- resolve the requested path against the selected workspace root
- normalize the resulting absolute path
- ensure it stays within the root directory boundary
- reject directory preview through the file-read API
- reject unsupported or oversized files safely

The renderer must never supply arbitrary absolute paths directly to the main process.

## Error Handling

Cases to handle explicitly:

- no workspace selected
- selected preview file deleted before refresh
- selected preview file replaced with an unsupported/binary file
- selected preview file exceeds preview size limit
- selected workspace folder becomes unavailable
- JSON parse failure

Expected behavior:

- show clear inspector state, not a crash
- preserve the rest of the app interaction model
- keep terminal nodes functional regardless of preview failures

## Testing And Verification

### Unit / Node-Level

Add tests for preview classification and guarded file reading:

- supported markdown file returns text preview payload
- valid JSON returns pretty-printed JSON payload
- invalid JSON still returns raw text payload
- unsupported extension returns fallback type
- oversized file returns `too-large`
- path traversal outside workspace root is rejected

### Smoke / Electron

Extend smoke coverage to verify:

- workspace tree still renders after opening a folder
- directory rows can expand/collapse
- selecting a supported file opens the right inspector
- manual refresh reloads changed file content
- unsupported file shows fallback state
- existing terminal/canvas smoke flow still passes

### Manual Verification

Manual check after implementation:

1. open a workspace folder
2. create or modify a markdown/json/text file from a terminal node
3. navigate to that file in `Folders`
4. open it in the inspector
5. verify contents are readable
6. modify the file again from the terminal
7. click `Refresh`
8. verify the inspector updates

## Acceptance Criteria

- user can expand/collapse folders in the left `Folders` section
- user can click markdown, JSON, and text/code files to preview them in a right inspector
- preview is read-only
- inspector has a manual refresh control
- unsupported or oversized files show a clear fallback state
- preview file access stays constrained to the selected workspace root
- terminal and canvas behavior remain unchanged

## Simplifying Decisions

To keep this version small and aligned with product intent:

- no editing
- no auto-refresh
- no markdown rendering engine
- no syntax highlighting dependency
- no canvas preview nodes
- no external file-tree library unless implementation proves the custom version is unexpectedly costly
